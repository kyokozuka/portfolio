# ADR-001: TypeScript strict モード採用

## Status
Accepted (2026-05-20)

## Context

ポートフォリオプロジェクトは JavaScript から TypeScript への移行を検討していました。

**課題**:
- 型安全性の欠如により、実行時エラーが頻発
- 推定月 30 時間のデバッグコストが発生
- チーム内でスキルレベルが異なり、コード品質が不均一

**選択肢**:
1. JSDoc による型注釈（TS なし）
2. TypeScript（`strict: false`）
3. TypeScript（`strict: true`）

## Decision

**TypeScript を `strict: true` で採用。**

すべてのコードに明示的な型定義を必須とし、型チェックを最大限に厳格化します。

## Consequences

### ✅ メリット

- **実行時エラー削減**: 推定 40% の実行時エラーを予防
- **開発体験向上**: IDE の自動補完とリファクタリング精度向上
- **チーム品質均一化**: 型定義により暗黙的な契約を明示化
- **ドキュメント効果**: 型定義自体が実装ドキュメントに

### ❌ デメリット

- **ビルド時間**: 10s → 30s に増加（3倍）
- **初期セットアップ**: 既存 JS コードの型定義付与に初期コスト
- **学習曲線**: TypeScript 未経験者の習熟期間が必要
- **開発速度**: 型定義による初期実装速度の低下

## Rationale

実行時エラー削減によるデバッグ時間短縮（月 30h × 40% = 月 12h）が、ビルド時間増加（月 15-20m 程度）を上回ると判断。

長期的な保守性とコード品質が最優先。

## Alternatives Considered

### 1. JSDoc による型注釈（採用せず）

**メリット**:
- TypeScript コンパイル不要
- 既存 JS コード資産の活用

**デメリット**:
- 実行時チェック不可（型定義は推奨のみ）
- IDE サポート限定的
- チーム教育コストが実質的に同等

**決定理由**: 実行時チェック機能の欠落が致命的。

### 2. TypeScript（`strict: false`）（採用せず）

**メリット**:
- グラデュアルな移行が可能
- 初期セットアップコストが低い

**デメリット**:
- 型安全性の効果が半減
- `any` の多用が増え、型定義の価値が低下
- チーム内で型定義品質が不均一

**決定理由**: 中途半端では効果が限定的。strict: true で統一することで初めて投資効果が出現。

## Implementation

- **実装期限**: 2026-05-20 ～
- **段階**:
  1. tsconfig.json を `strict: true` で設定
  2. 既存 JS ファイルを TS に移行
  3. 型定義を充実化
