# ADR-002: フィーチャーベース + レイヤード アーキテクチャ

## Status
Accepted (2026-05-20)

## Context

ポートフォリオプロジェクトの複数ページ（home, software, uiux, contact）を開発するにあたり、スケーラブルで保守性の高いアーキテクチャが必要でした。

**課題**:
- ページ数増加に伴い、共有コンポーネントと独立コンポーネントの境界が曖昧
- クロスフィーチャーの不要な依存関係が増加する懸念
- UI、ロジック、データの関心の分離が不明確

## Decision

**フィーチャーベース + レイヤード アーキテクチャを採用。**

### 構造

```
src/
├── app/              ← ルーティング層（薄い）
├── features/         ← フィーチャーモジュール
│   ├── [feature]/
│   │   ├── ui/       ← UI コンポーネント
│   │   ├── model/    ← ロジック（フック）
│   │   ├── data/     ← データ・定数
│   │   ├── types/    ← 型定義
│   │   └── README.md
└── shared/           ← 共有コード（2+ フィーチャー）
    ├── ui/
    ├── data/
    └── hooks/
```

### インポート方向

```
app/ → features/ → shared/
↑ (never go up)
```

## Consequences

### ✅ メリット

- **スケーラビリティ**: 新機能追加時、既存フィーチャーへの影響が最小化
- **保守性**: 各フィーチャーが独立しており、変更の影響が局所化
- **テスト容易性**: フィーチャー単位でのテストが可能
- **チーム協業**: 複数チーム間でのコンフリクト最小化
- **リファクタリング容易**: 関連コードが一箇所に集中

### ❌ デメリット

- **初期オーバーヘッド**: ファイル・フォルダ構造の管理コスト
- **共有判定の難しさ**: 2+ フィーチャー共有か、フィーチャー独立か判断が必要
- **小規模プロジェクトでは過度設計**: 現在のポートフォリオ規模では若干のオーバーヘッド

## Rationale

現在 4 フィーチャー（home, software, uiux, contact）ですが、将来的な拡張（プロジェクト追加、スキル詳細ページ、ブログなど）を想定すると、スケーラブルな設計が投資効果を生み出します。

初期セットアップコスト < 将来的な保守コスト削減。

## Alternatives Considered

### 1. MVC（Model-View-Controller）（採用せず）

**メリット**: 伝統的で理解しやすい

**デメリット**: 
- Model が肥大化しやすい
- ページ追加ごとに新しい Controller を作成する必要

**決定理由**: フロントエンドでの MVC は React の設計思想と合わないため、フィーチャーベースが自然。

### 2. 単一フォルダ（すべてを `src/` に）（採用せず）

**メリット**: シンプル

**デメリット**: 
- ファイル数増加で混乱
- 共有コンポーネントと独立コンポーネントが混在

**決定理由**: スケーラビリティ重視により不採用。

## Implementation

- **実装期限**: 2026-05-20 ～
- **段階**: 既存フィーチャー（home, software, uiux, contact）を構造に整列
