# ADR-004: セマンティックHTML必須化

## Status
Accepted (2026-05-20)

## Context

ポートフォリオプロジェクトでコンポーネントを実装する際、`<div>` で構造を実装するか、セマンティックな HTML 要素（`<header>`, `<nav>`, `<section>`, `<article>` など）を使用するか決定が必要でした。

**課題**:
- スクリーンリーダーユーザーへのアクセシビリティ
- SEO での構造化データの価値
- コード保守性（意図が明確か）
- チーム開発での一貫性

## Decision

**セマンティックHTML要素の使用を必須化。**

ジェネリックな `<div>` + CSS クラスではなく、意図を反映した HTML 要素を常に使用する。

### 必須要素マッピング

| 用途 | セマンティック要素 | 代わりに使用しない |
|-----|------------------|-----------------|
| ページレベルナビゲーション | `<header>`, `<nav>`, `<footer>` | `<div class="header">` |
| メインコンテンツ | `<main>` | `<div class="main-content">` |
| セクション | `<section aria-labelledby="...">` | `<div class="section">` |
| 自己完結コンテンツ | `<article>` | `<div class="card">` |
| リスト | `<ul>` + `<li>` | `<div class="list">` |
| 補完情報 | `<aside>` | `<div class="sidebar">` |

### 許容される `<div>` / `<span>` の用途

`<div>` / `<span>` は「非モダン」ではなく、**意味を持たない純粋なレイアウト・装飾のまとまり**には正しく使う。判断は次の一文に集約する。

> **意味があればセマンティック要素、意味がなくスタイル目的だけなら `<div>` / `<span>`。**

- ✅ 許容: グリッド/フレックスのラッパー、余白・境界のためのボックス、アニメーション用ラッパー（`reveal-item` 等）、インラインの装飾スパン。
- ❌ 非許容: `section` / `article` / `nav` / `header` / `footer` / `main` / `address` / `ul` 等で表せる箇所を `<div>` で代替すること（div スープ）。
- 💡 不要な中間ラッパーは **React Fragment（`<>...</>`）** に置き換え、要素数自体を増やさない。CSS Grid/Flex でラッパーを減らせないかも検討する。

つまり本ルールは「`<div>` 禁止」ではなく「**セマンティック要素を優先し、`<div>` は非意味のレイアウトに限定する**」である。

## Consequences

### ✅ メリット

- **アクセシビリティ**: スクリーンリーダーが構造を理解可能
- **SEO**: 検索エンジンが HTML 構造から意図を抽出
- **メンテナンス性**: コードの意図が明示的で、将来のリファクタリングが容易
- **WCAG 準拠**: Web Content Accessibility Guidelines に準拠
- **チーム統一**: 実装ルールが明確で、コードレビューが簡潔

### ❌ デメリット

- **学習コスト**: HTML5 セマンティック要素の知識が必要
- **柔軟性の制約**: CSS での微調整が限定されることがある
- **ネストの複雑化**: 正しい階層構造を保つためのネストが深くなる可能性

## Rationale

ポートフォリオは「個人の専門性を示すもの」であり、アクセシビリティと SEO はブランドイメージに直結します。

初期学習コスト < 長期的なメンテナンス利益 + ブランド価値向上。

## Alternatives Considered

### 1. すべて `<div>` + CSS クラス（採用せず）

**メリット**: 
- CSS での自由度が高い
- 学習曲線が低い

**デメリット**: 
- アクセシビリティが低下
- SEO での構造化が困難
- 意図がコードから不明確

**決定理由**: ポートフォリオとしてのアクセシビリティは必須。

### 2. BEM（Block Element Modifier）（採用せず、ただし CSS では採用可能）

**メリット**: 
- CSS クラス命名が体系的

**デメリット**: 
- HTML 要素の選択とは独立した議論
- セマンティック HTML と組み合わせるべき

**決定理由**: BEM と セマンティック HTML は相補的（共存可能）。
