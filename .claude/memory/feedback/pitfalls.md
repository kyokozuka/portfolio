# 開発の落とし穴と制約

## アーキテクチャ決定

- **TypeScript strict モードは譲歩できない**
  - **理由:** 型安全性がコード品質の基盤；`any` 型は全体システムを損なう
  - **適用方法:** `strict: false`、段階的移行、JSDoc のみのアプローチは決して提案しない

- **フィーチャーベース + レイヤード構造でスコープ膨張を防ぐ**
  - **理由:** 過去プロジェクトで共有コンポーネント境界が曖昧になり、循環依存が増加
  - **適用方法:** 機能追加時は、責務を明確に（feature ディレクトリ vs shared/）

- **セマンティック HTML は必須、オプションではない**
  - **理由:** アクセシビリティと SEO がポートフォリオのブランド価値；`<div>` soup は専門性を損なう
  - **適用方法:** コンポーネント レビュー時にセマンティック構造を確認

## デザインシステム制約

- **Signal/Noise システムは厳密に設計されている**
  - **理由:** 単一のアクセントカラー（`#C8552A`）が視覚的アイデンティティ全体；色を増やすと効果が薄れる
  - **適用方法:** 2番目のアクセント色や追加色の誘惑に駆られたら、Signal/Noise の根拠を再検討

- **CSS-in-JS のショートカットは避ける**
  - **理由:** Tailwind + CSS custom properties が必要な制約を提供；ランタイムオーバーヘッドは不要
  - **適用方法:** Tailwind ユーティリティ + `:root` カスタムプロパティを優先；styled-components/Emotion はスコープ外

## ドキュメント アンチパターン

- **重複はメンテナンス負担を生み出す**
  - **理由:** 2つのデザインドキュメントは時間とともに乖離；貢献者が間違ったファイルを更新する
  - **適用方法:** トピックごとに唯一の情報源（例：`docs/design.md` のみ；DESIGN.md + docs/design.md ではない）

- **ADR 決定は再討議しない**
  - **理由:** ADR-001～005 は受理済み；再開は時間の無駄
  - **適用方法:** コード内のコメントは ADR を参照可能；新しいアーキテクチャ課題には新 ADR を作成（修正ではなく）

## パフォーマンス・品質

- **テストカバレッジとパフォーマンスは実装の明確性に次ぐ**
  - **理由:** ポートフォリオサイトで高トラフィック製品ではない；クリーンで十分にドキュメント化された体験を出荷することが重要
  - **適用方法:** テストやキャッシングを過度に設計しない；クリアなコードに注力
